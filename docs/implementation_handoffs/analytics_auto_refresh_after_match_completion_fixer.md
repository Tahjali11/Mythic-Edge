## Finding Fixed

`CT-294-001 P1`: persisted timestamp-like SQLite metadata could echo unsafe strings through `GET /api/analytics/refresh-state`.

Codex E also noted that the same unsafe timestamp-like metadata could be included in the opaque revision input, and that frontend refresh-state validation accepted arbitrary strings for timestamp fields.

## Fix Summary

- Added a focused backend regression proving unsafe persisted timestamp-like SQLite metadata is not echoed through public refresh-state fields.
- Backend refresh-state timestamp reads now accept only parseable ISO-like timestamps; unsafe values become `null`.
- Backend revision hashing now uses sanitized table timestamp metadata and sanitized latest-completed timestamp inputs only.
- Added a frontend API regression for unsafe refresh-state timestamp fields.
- Frontend refresh-state validation now requires `latest_completed_match_seen_at` and `latest_completed_ingest_finished_at` to be ISO timestamp strings or `null`.
